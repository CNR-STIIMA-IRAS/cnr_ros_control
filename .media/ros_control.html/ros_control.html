ROScontrolframework
The ROS control framework originates from the PR2
robot control package. Currently, the framework
includes several packages under the name of
ros_control.
The idea behind ros_control is to allow easy
development of controllers with standardized code for
all robots supporting ROS ecosystem, aiming to:
 facilitating the quick and effective use of
controllers/robots code not written by you
 implementing controllers / robots written by you and
make them easily usable quickly to other developers
ros_control permits out-of-the box compatibility with
third party tools, in this way:
 Promote reuse of control code
 Provide ready-to-use tools
Slide 3
ROScontrolframework
The primary objects of the framework, as well as those
most noteworthy, are 3 core classes:
 RobotHardware (RobotHW): software representation
of the robot, it is the the first layer of code that
standardize your real robot driver.
 The instantiation of the class is in a ROS-node that
bringup communication with the hardware. Functions
read() and write() allow to interact with driver. It must
be designed ad-hoc for the driver.
 HardwareInterface: an interface to access set of
similar resources for one or multiple joints. (e.g. joint
velocity). Typical ros HardwareInterface:
-Read-only: JointState / IMU / Force-torque sensor
-Read-write: Position joint / Velocity joint / Effort joint
 JointHandle: a class that handles a joint name with
related resources.
